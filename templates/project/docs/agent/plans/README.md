# 方案落实台账目录

本目录用于保存 `<项目名>` 中已经从讨论整理为开发顺序的方案落实台账。

方案落实台账不是原始会话记录，也不是当前事实源。它是方案从讨论走向开发的执行合同，用于防止多轮讨论中的确认点在实现时丢失。

## 文件命名

```text
docs/agent/plans/<plan-id>.md
```

建议使用稳定 ID：

```text
plan-YYYYMMDD-短名称.md
```

## 状态

| 状态 | 含义 | 使用规则 |
|---|---|---|
| `draft` | 根据讨论整理出的草稿 | 不能直接作为开发依据 |
| `confirmed` | 用户确认过的定稿 | 可作为实现依据 |
| `active` | 正在按序开发 | 必须维护任务状态 |
| `done` | 所有任务已完成或有明确处理结论 | 可触发最终文档同步 |
| `superseded` | 已被新台账替代 | 只用于回溯 |

## 任务状态

| 状态 | 含义 |
|---|---|
| `pending` | 待开始 |
| `in_progress` | 正在处理 |
| `done` | 已完成并关联验证 |
| `blocked` | 被阻塞，需要确认或外部条件 |
| `deferred` | 用户确认本轮延期或不做 |

## 使用规则

- 只有用户要求形成开发顺序、任务清单、记录草稿或准备落地时，才创建 `draft` 台账。
- `draft` 台账不得作为实现依据。
- 用户确认后，台账状态才能改为 `confirmed`。
- 用户要求按完整方案开发时，Agent 必须读取 `confirmed` 或 `active` 台账。
- 完整方案开发必须按开发顺序逐项执行，不能静默跳过 `pending`、`blocked` 或 `open` 项。
- 台账完成后，再评估是否把稳定事实写回 `memory.md`、业务文档、模块文档、决策记录或项目风格画像。
