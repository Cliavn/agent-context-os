# Workflow：方案理解与摄取

## 目标

当用户输入完整方案、业务计划、产品计划、技术方案或执行计划时，先理解方案初衷和业务意义，再拆解实现逻辑、系统影响和融合更新项。

## 步骤

1. 确认方案来源、类型、适用范围和用户希望达成的目标。
2. 阅读 `docs/agent/intake.md`、`docs/agent/memory.md`、相关业务文档和模块文档。
3. 先提取方案意图：业务初衷、痛点、成功标准和非目标。
4. 再提取业务意义：用户角色、业务对象、流程、字段、状态、异常和验收口径。
5. 分析实现逻辑：模块、接口、数据、权限、配置、运行态、验证和项目风格影响。
6. 生成系统影响矩阵。
7. 检查与当前记忆、旧文档、代码事实或测试结果是否冲突。
8. 将方案内容标记为 `current`、`proposed`、`assumption`、`open` 或 `conflict`。
9. 将有效内容写回 `intake.md`、`memory.md`、`style-profile.md`、业务文档、模块文档、决策记录或任务报告。
10. 对未确认内容生成开放问题。
11. 输出方案摄取报告。

## 禁止事项

- 不允许未理解业务初衷就直接拆任务。
- 不允许只把方案全文保存为单文件。
- 不允许把方案建议直接覆盖当前事实。
- 不允许忽略连带功能影响。
- 不允许让后续任务默认读取方案全文。
