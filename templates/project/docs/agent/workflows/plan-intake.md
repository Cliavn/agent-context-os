# Workflow：方案理解与摄取

## 目标

当用户输入完整方案、业务计划、产品计划、技术方案或执行计划时，先理解方案初衷和业务意义，再拆解实现逻辑、系统影响和融合更新项。

## 步骤

1. 判断当前是讨论阶段（`discussion_only`）、草案记录（`draft_record`）、定版摄取还是执行落地。
2. 如果用户只是讨论、比较或推敲方案，只在对话中整理候选方案、取舍、风险和待确认问题，不修改协作文档。
3. 如果用户要求形成开发顺序、任务清单、记录草稿或准备落地，创建或更新 `docs/agent/plans/<plan-id>.md`，状态为 `draft`。
4. 获得用户对定版、执行、记录草案或写回的明确确认后，确认方案来源、类型、适用范围和用户希望达成的目标。
5. 阅读 `docs/agent/intake.md`、`docs/agent/plans/README.md`、`docs/agent/memory.md`、相关业务文档和模块文档。
6. 先提取方案意图：业务初衷、痛点、成功标准和非目标。
7. 再提取业务意义：用户角色、业务对象、流程、字段、状态、异常和验收口径。
8. 分析实现逻辑：模块、接口、数据、权限、配置、运行态、验证和项目风格影响。
9. 生成系统影响矩阵。
10. 将确认点整理为开发顺序和任务清单，每个任务必须有任务 ID、来源摘要、依赖、验收口径和状态。
11. 检查与当前记忆、旧文档、代码事实或测试结果是否冲突。
12. 将方案内容标记为 `current`、`proposed`、`assumption`、`open` 或 `conflict`。
13. 按确认范围写回 `intake.md`、`plans/<plan-id>.md`、`memory.md`、`style-profile.md`、业务文档、模块文档、决策记录或任务报告；记录草案时只能写入 `proposed`、`open` 或 `assumption`，不得覆盖 `current`。
14. 对未确认内容生成开放问题。
15. 输出方案摄取报告。

## 禁止事项

- 不允许未理解业务初衷就直接拆任务。
- 不允许只把方案全文保存为单文件。
- 不允许在方案讨论阶段修改协作文档。
- 不允许把原始会话全文保存为方案落实台账。
- 不允许把 `draft` 台账作为开发依据。
- 不允许把方案建议直接覆盖当前事实。
- 不允许忽略连带功能影响。
- 不允许让后续任务默认读取方案全文。
