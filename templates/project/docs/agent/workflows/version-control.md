# Workflow：版本控制

## 目标

版本控制操作必须可审计、范围清晰，并避免提交无关改动。

完成代码修改并通过必要验证后，默认创建本地 Git 提交。推送到远端仓库仍必须获得用户明确授权。

最终回复结尾必须包含本地提交状态和“是否推送”；即使没有创建提交，也必须明确说明是否已推送。

## 步骤

1. 阅读入口、质量规则和本 workflow。
2. 检查工作区状态。
3. 区分本次改动和用户已有改动。
4. 运行必要验证。
5. 只暂存本次相关文件。
6. 使用清晰提交信息。
7. 创建本地 Git 提交。
8. 记录是否推送；推送前获得用户明确授权。
9. 在最终回复结尾输出本地提交状态和“是否推送”。

## 可跳过本地提交的情况

- 用户明确要求本次不要提交。
- 本次没有产生文件变更。
- 验证失败且用户没有明确接受失败状态。
- 无法安全区分本次 Agent 改动和用户已有改动。
- 当前目录不是 Git 仓库，或仓库状态阻止提交。

## 禁止事项

- 不允许提交未确认的无关改动。
- 不允许在验证失败后假装通过。
- 不允许把用户已有未提交改动混入本次提交。
- 不允许未授权推送到远端。
- 不允许省略最终回复中的“是否推送”。
- 不允许使用破坏性命令清理用户改动。
