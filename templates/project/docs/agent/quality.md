# 质量门禁

## 代码质量

- 修改范围必须与需求匹配。
- 优先复用现有结构、命名、接口和工具链。
- 不引入无必要的新抽象。
- 错误处理必须可观察、可恢复或可明确失败。
- 状态、缓存、并发、文件系统、网络和数据库逻辑必须有明确生命周期。

## 测试标准

- 新增业务逻辑优先补单元测试。
- 修复 Bug 优先补回归测试。
- 跨模块流程补集成测试或手工验收。
- 重构必须证明外部行为不变。
- 无法测试时必须说明原因、手工验证方式和剩余风险。

## 完成定义

一次任务完成必须满足：

- 已阅读相关上下文。
- 已遵守模块边界。
- 已执行或说明无法执行的验证。
- 已评估文档同步需求。
- 已在最终报告中说明风险。
