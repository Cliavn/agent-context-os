# 检索式项目记忆

## 目标

检索式项目记忆用于让 Agent 快速找到与当前任务相关的项目记忆、代码路径和证据位置。它不替代项目记忆源，也不替代代码、测试或用户确认。

新版模型采用：

```text
项目记忆源 + 真实代码 / 测试
  ↓
本地索引
  ↓
Agent 召回摘要和路径
  ↓
Agent 深读真实事实源
```

## 关键边界

- 项目记忆源进入 Git（版本控制工具）。
- 本地索引不进入 Git。
- 向量数据库文件不是团队事实源。
- 索引可以由记忆源和代码证据重建。
- 检索命中只是线索，不是最终判断。

## 推荐存储职责

| 内容 | 是否进入 Git | 说明 |
|---|---|---|
| 项目记忆源 | 是 | 小型 Markdown / JSONL，团队可审查 |
| 代码和测试 | 是 | 最高事实优先级 |
| 本地向量索引 | 否 | 可重建产物，放本机缓存或 `.agent-context/local-index/` |
| 全文 / 关键词索引 | 否 | 可重建产物 |
| 索引 manifest | 可选 | 只保存版本、模型、hash 和重建信息 |

## 索引记录内容

向量库或本地索引不应只存路径，也不应存不可审查的大段正文。推荐保存：

- 短摘要。
- 关键词。
- 模块、业务对象、接口名或文件路径。
- `status`、`type`、`scope`、`confidence`、`last_verified`。
- 证据路径。
- 用于相似度搜索的 embedding（嵌入向量）。

## 最小数据模型

项目记忆源中的每条记录必须可追溯：

```json
{
  "id": "mem-YYYYMMDD-001",
  "status": "current",
  "type": "business_rule",
  "scope": ["<模块名>", "<业务对象>"],
  "summary": "<一句话摘要>",
  "source": {
    "kind": "user_confirmed",
    "ref": "<来源路径或说明>",
    "date": "YYYY-MM-DD"
  },
  "evidence": ["<代码路径或测试路径>"],
  "confidence": "high",
  "last_verified": "YYYY-MM-DD",
  "tags": ["<标签>"]
}
```

本地索引可以在此基础上增加 `vector`、`keywords`、`indexed_at`、`source_hash` 和 `engine_version` 等字段。

## 检索策略

检索应采用混合策略，而不是只依赖向量相似度：

1. 关键词检索：匹配模块名、接口名、字段名、页面名、业务对象和错误码。
2. 结构化过滤：按 `project_id`、`status`、`type`、`scope`、`tags`、`last_verified` 筛选。
3. 向量检索：召回语义相近的业务场景、历史坑和决策背景。
4. 新鲜度排序：优先使用最近校验且仍为 `current` 的记忆。
5. 证据优先：用户确认、代码、测试和当前文档优先于对话摘要。

## 默认召回

| 任务 | 默认召回 |
|---|---|
| 新增功能 | `business_rule`、`interaction_rule`、`architecture_rule`、`decision` |
| 修复 Bug | `known_issue`、`implementation_note`、相关 `business_rule` |
| UI 改造 | `interaction_rule`、风格事实、相关业务规则 |
| 数据库或接口变更 | `architecture_rule`、`implementation_note`、字段和接口相关记忆 |
| 方案摄取 | `decision`、`open_question`、冲突范围内的当前事实 |

## 写入与重建

1. 从对话、任务报告、代码变更、文档变更或用户确认中识别可复用事实。
2. 优先写入项目记忆源。
3. 记录来源、证据、可信度和最后校验日期。
4. 如果信息不完整，写为 `draft` 或 `open_question`，不得标为 `current`。
5. 如果只是 Agent 推测，最多写为 `assumption`，并设置低可信度。
6. 标记本地索引需要增量更新或重建。
7. 后续发现冲突时，不删除历史记录；标记为 `stale` 或 `deprecated` 并指向新事实。

## 读取流程

任务开始时，Agent 应先判断是否需要检索项目记忆：

- `S0` 微小变更默认不检索，除非用户补充业务背景或涉及风险点。
- `S1` 按模块、文件路径、任务类型做小范围检索。
- `S2` 必须检索相关业务、架构、实现和历史问题记忆。
- `S3` 必须检索决策、历史问题、开放问题和冲突事实。

召回后必须做三件事：

1. 只采用与当前任务相关的记忆。
2. 对高风险或过期记忆回到代码、测试或用户确认验证。
3. 在任务报告中说明使用了哪些记忆，以及是否发现过期或冲突。

## 嵌入式与服务型

默认推荐本地嵌入式索引，降低用户部署成本。协作引擎应通过 provider（提供器）接口访问索引，保留未来切换到本地服务型或团队服务型检索后端的能力。

无论底层是嵌入式还是服务型，团队共享事实都应以项目记忆源为准。

## 隐私和安全

- 不得写入账号、密钥、token、cookie 或真实隐私数据。
- 不得写入无法脱敏的客户数据、生产数据或安全凭据。
- 对外部系统中的 Issue、PR 或任务链接，只记录必要引用，不复制敏感正文。
- 记忆内容应按最小必要原则记录，避免把一次性对话完整存档。

## 质量门禁

启用检索式项目记忆后，任务完成前必须回答：

- 本次是否新增、更新、废弃或校验了项目记忆源。
- 本地索引是否需要重建或增量更新。
- 召回记忆是否影响实现方案。
- 是否发现过期、冲突或低可信记忆。
- 是否已回到代码、测试或用户确认验证高风险内容。
