# 项目记忆系统

## 目标

项目记忆系统用于把用户在协作过程中补充的业务背景、实现约束、历史坑和验收口径，沉淀为可继承、可路由、可验证的项目事实。

Agent 不应只在当前对话中临时使用这些信息，而应判断它们是否需要写回项目上下文。但处于讨论、比较或推敲阶段的方案方向，不得因为“可复用”就自动写回；必须先获得用户对定版、执行、记录草案或写回的明确确认。

当项目记忆规模变大时，应启用检索式项目记忆存储。它负责快速召回相关记忆，当前事实仍应维护在人类可审查的文档中。完整规范见 `docs/14-retrieval-memory-store.md`。

## 记忆类型

| 类型 | 说明 | 推荐位置 |
|---|---|---|
| 项目事实 | 项目目标、非目标、用户、阶段、风险 | `docs/agent/01-project-overview.md` |
| 业务事实 | 业务对象、流程、字段、状态、异常、验收 | 业务语义文档或 `docs/agent/memory.md` |
| 风格事实 | 前端、后端、客户端等端侧风格、交互习惯、命名偏好和工程约定 | `docs/agent/style-profile.md` |
| 架构事实 | 模块职责、依赖方向、禁止事项、关键取舍 | `docs/agent/02-architecture.md`、`docs/agent/04-decisions.md`、模块文档 |
| 实现事实 | 启动方式、验证命令、技术栈、运行限制 | `docs/agent/03-tech-stack.md`、模块文档 |
| 历史问题 | Bug 现象、根因、防复发规则、回归验证 | 历史问题库或任务报告 |
| 开放问题 | 用户尚未确认、Agent 不能安全假设的信息 | `docs/agent/memory.md` |

## 检索式记忆

模板项目可以使用 `docs/agent/memory-store/` 保存结构化检索记忆：

```text
docs/agent/memory-store/
├── README.md
├── memory-schema.json
├── memories.jsonl
└── retrieval-config.json
```

检索式记忆用于快速定位相关上下文，不是唯一事实来源。事实优先级为：

```text
代码和测试 > 当前文档 > 决策记录 > 检索记忆 > 对话摘要
```

写入检索记忆时，必须记录 `status`、`type`、`scope`、`summary`、`content`、`source`、`confidence` 和 `last_verified`。

读取检索记忆时，必须结合任务类型、模块、文件路径、接口名、页面名或业务对象进行筛选。高风险、过期或低可信记忆必须回到代码、测试或当前文档验证。

## 事实状态

项目记忆必须区分事实、草案和推测：

| 状态 | 含义 | 使用规则 |
|---|---|---|
| `current` | 已确认的当前事实 | 可作为实现依据 |
| `draft` | 用户表达了方向，但尚未完整确认 | 使用前需要确认关键细节 |
| `assumption` | Agent 为推进任务做出的假设 | 不得沉淀为当前事实，必须等待确认 |
| `deprecated` | 已废弃或被新规则替代 | 只用于回溯，不作为实现依据 |
| `stale` | 疑似过期或与当前实现冲突 | 使用前必须重新验证 |

## 记忆回写触发条件

出现以下情况时，Agent 必须评估是否更新项目记忆：

- 用户补充了新的业务背景、业务规则、用户角色或验收口径。
- 用户补充了新的项目风格、交互习惯、命名偏好、端侧工程约定或不希望重复解释的实现习惯。
- 用户纠正了 Agent 对业务、架构或实现的理解。
- 任务新增或改变了字段、状态、接口、权限、流程、模块边界或验证命令。
- 修复了重复 Bug，或发现现有 checklist 无法防止同类问题。
- Agent 在实现过程中发现文档与代码或测试结果不一致。
- 用户表达了偏好、命名习惯、边界条件或不希望再重复解释的上下文。

## 回写流程

1. 识别本次对话或任务中是否出现可复用项目事实。
2. 判断事实类型、状态和适用范围。
3. 判断该信息是否已被用户确认为当前事实；如果仍是方案讨论，只在对话中整理，不修改协作文档。
4. 如果信息完整且已确认，写入对应项目文档。
5. 如果启用了检索式记忆，同步写入或更新 `docs/agent/memory-store/memories.jsonl`，并记录来源、证据、可信度和最后校验日期。
6. 如果信息不完整，写入开放问题，并在当前任务中主动追问。
7. 如果只是临时假设，只记录为 `assumption`，不得覆盖 `current`。
8. 在任务报告中说明记忆回写结果。

## 主动引导

Agent 发现业务信息不足时，不应等用户手动提醒，应主动提出最小必要问题。

优先追问以下内容：

- 这个规则适用于哪些用户、模块或场景。
- 成功和失败分别如何判断。
- 是否存在历史例外、人工处理点或不可逆操作。
- 字段、状态或流程是否影响已有数据。
- 该信息应作为当前事实、草案还是待确认问题。

## 防误记规则

- 不得把 Agent 推测写成 `current`。
- 不得在方案讨论阶段把未定版方向写入当前协作文档。
- 不得把一次性临时需求提升为通用业务规则。
- 不得记录账号、密钥、token、cookie 或真实隐私数据。
- 不得在没有适用范围的情况下覆盖已有事实。
- 文档与代码冲突时，应优先引用代码和测试结果，并标记文档需要更新。
- 检索记忆与代码、测试或当前文档冲突时，不得直接删除历史记录；应标记为 `stale` 或 `deprecated`，并指向新的事实源。

## 质量门禁

任务完成前必须回答：

- 本次是否出现新的可复用业务或项目事实。
- 是否需要更新项目记忆、项目风格画像、业务文档、模块文档、历史问题库或 checklist。
- 如果没有更新，原因是什么。
- 是否存在需要用户确认的开放问题。
- 如果启用了 `docs/agent/memory-store/`，是否需要新增、更新、废弃或校验检索记忆。
