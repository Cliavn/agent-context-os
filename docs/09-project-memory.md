# 项目记忆系统

## 目标

项目记忆系统用于沉淀用户项目中可复用、可审查、可追溯的事实，让 Agent 在后续任务中不用反复向用户确认同一批业务背景、开发习惯和历史坑。

项目记忆不等于协作引擎规范。协作引擎规范定义 Agent 怎么工作；项目记忆定义这个项目是什么、业务怎么跑、团队怎么开发、用户怎么使用。

## 记忆三分法

| 类型 | 归属 | 示例 | 是否进入项目记忆源 |
|---|---|---|---|
| 协作引擎规范 | Agent Context OS | S0-S3、上下文路由、任务拆解、质量门禁、多 Agent 策略 | 否 |
| 项目共享记忆 | 用户项目 | 业务规则、交互习惯、开发习惯、模块边界、历史坑、验收口径 | 是 |
| 个人偏好记忆 | 用户个人环境 | 回复风格、个人工作偏好、个人常用命令 | 默认否 |

## 项目记忆类型

| 类型 | 说明 |
|---|---|
| `project_fact` | 项目目标、非目标、用户、阶段、风险 |
| `business_rule` | 业务对象、流程、字段、状态、异常、验收口径 |
| `interaction_rule` | 用户使用习惯、页面交互、端侧风格、文案偏好 |
| `architecture_rule` | 模块职责、依赖方向、禁止事项、关键取舍 |
| `implementation_note` | 启动方式、验证命令、技术限制、实现约束 |
| `known_issue` | 历史问题、根因、防复发规则、回归验证 |
| `decision` | 已确认的产品、技术或协作决策 |
| `open_question` | 用户尚未确认、Agent 不能安全假设的信息 |

## 项目记忆源

项目记忆源是团队共享事实，推荐放在：

```text
.agent-context/memory-sources/
```

它应使用小型 Markdown 或 JSONL 文件，进入 Git（版本控制工具），便于评审、合并、回滚和审计。

最小 JSONL 记录：

```json
{
  "id": "mem-YYYYMMDD-001",
  "status": "current",
  "type": "business_rule",
  "scope": ["<模块名>", "<业务对象>"],
  "summary": "<一句话摘要>",
  "source": {
    "kind": "user_confirmed",
    "ref": "<来源路径或说明>",
    "date": "YYYY-MM-DD"
  },
  "evidence": ["<代码路径或测试路径>"],
  "confidence": "high",
  "last_verified": "YYYY-MM-DD",
  "tags": ["<标签>"]
}
```

## 本地索引关系

本地向量库、全文索引或关键词索引不是项目记忆本体。它只由项目记忆源和代码证据生成，用于快速召回：

```text
项目记忆源
  ↓
本地检索索引
  ↓
摘要、模块、路径、证据
  ↓
Agent 阅读真实代码、测试或文档
```

本地索引可以删除并重建，必须被 `.gitignore` 排除。

## 事实状态

项目记忆必须区分事实、草案和推测：

| 状态 | 含义 | 使用规则 |
|---|---|---|
| `current` | 已确认的当前事实 | 可作为实现依据 |
| `draft` | 用户表达了方向，但尚未完整确认 | 使用前需要确认关键细节 |
| `assumption` | Agent 为推进任务做出的假设 | 不得沉淀为当前事实，必须等待确认 |
| `deprecated` | 已废弃或被新规则替代 | 只用于回溯，不作为实现依据 |
| `stale` | 疑似过期或与当前实现冲突 | 使用前必须重新验证 |

## 写回触发

出现以下情况时，Agent 必须评估是否更新项目记忆源：

- 用户补充了新的业务背景、业务规则、用户角色或验收口径。
- 用户补充了新的项目风格、交互习惯、命名偏好或端侧工程约定。
- 用户纠正了 Agent 对业务、架构或实现的理解。
- 任务新增或改变了字段、状态、接口、权限、流程、模块边界或验证命令。
- 修复了重复 Bug，或发现现有规则无法防止同类问题。
- Agent 在实现过程中发现项目记忆与代码或测试结果不一致。
- 用户表达了不希望再重复解释的上下文。

## 写回流程

1. 识别本次对话或任务中是否出现可复用项目事实。
2. 判断事实类型、状态、适用范围和证据路径。
3. 如果仍是方案讨论，只在对话中整理，不写入项目记忆源。
4. 如果信息完整且已确认，写入项目记忆源。
5. 如果本地索引已启用，标记需要重建或增量更新索引。
6. 如果信息不完整，写入开放问题或在当前任务中追问。
7. 如果只是临时假设，只记录为 `assumption`，不得覆盖 `current`。
8. 在任务报告中说明记忆源写回和本地索引状态。

## 防误记规则

- 不得把 Agent 推测写成 `current`。
- 不得把协作引擎规范写入用户项目记忆源。
- 不得在方案讨论阶段把未定版方向写入当前事实。
- 不得把一次性临时需求提升为通用业务规则。
- 不得记录账号、密钥、token、cookie 或真实隐私数据。
- 不得在没有适用范围和证据路径的情况下覆盖已有事实。
- 项目记忆与代码、测试冲突时，应优先引用代码和测试结果，并将记忆标记为 `stale` 或更新为新事实。

## 质量门禁

任务完成前必须回答：

- 本次是否出现新的可复用业务或项目事实。
- 是否需要更新项目记忆源。
- 是否需要重建或增量更新本地检索索引。
- 如果没有更新，原因是什么。
- 是否存在需要用户确认的开放问题。
