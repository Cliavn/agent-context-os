# 架构边界

## 目标

架构边界用于让 Agent 在修改代码前理解模块职责、依赖方向和禁止事项，避免跨模块乱改。

## 模块卡片

每个关键模块应有清晰的边界记录。薄入口项目默认把模块边界沉淀为项目记忆源，而不是重新创建旧版协作引擎目录：

```text
.agent-context/memory-sources/memory-architecture.jsonl
```

如果项目本身已经有架构文档，可以在记忆源的 `source.ref` 或 `evidence` 中指向真实文档路径。不要为了升级协作引擎而新建 `docs/agent/modules/`。

模块边界记录应包含：

| 部分 | 说明 |
|---|---|
| 职责 | 模块负责什么 |
| 不负责 | 模块明确不做什么 |
| 允许依赖 | 可以调用哪些模块或接口 |
| 禁止依赖 | 不能绕过哪些边界 |
| 主要入口 | 关键代码路径 |
| 数据流 | 输入、输出、状态变化 |
| 验证方式 | 修改后应执行哪些检查 |
| 常见风险 | 历史坑、易错点、并发或缓存风险 |

## 依赖原则

- 高层模块通过公开接口依赖底层能力。
- UI 不应直接绕过 API 或服务层修改持久化数据。
- 自动化执行模块不应直接承担任务编排职责。
- 存储模块负责 schema、migration 和 repository 边界。
- 安全模块负责凭据、token、cookie 和隐私数据处理规则。

## 边界冲突处理

如果任务需要突破现有边界，Agent 必须先输出：

1. 为什么现有边界不足。
2. 替代方案有哪些。
3. 影响哪些模块。
4. 是否需要更新架构文档。
5. 如何回滚。
