# 执行门禁与变更分级

## 目标

执行门禁与变更分级用于保证 Agent Context Engine 既能约束高风险变更，又不会让微小改动承担过重成本。

核心目标：

- 小事轻：微小样式、文案和局部调整走轻量模式。
- 大事稳：业务、模块、方案、架构和高风险变更走完整流程。
- 不偏移：代码修改后必须评估项目记忆源、本地索引、业务文档和模块文档是否需要同步。
- 可检查：不依赖 Agent 自觉，关键规则应能被脚本、提交门禁或 CI 检查。
- 可并行：多个 Codex 任务并行开发时，通过独立 Git worktree（工作树）隔离文件系统和 Git index（暂存区）。

## 变更等级

| 等级 | 名称 | 示例 | 默认流程 |
|---|---|---|---|
| `S0` | 微小变更 | 按钮颜色、间距、错别字、纯样式 | 轻量模式 |
| `S1` | 局部变更 | 单组件交互、小范围 Bug、局部逻辑 | 标准局部流程 |
| `S2` | 业务或模块变更 | 字段、状态、接口、权限、模块边界 | 完整记忆源和验证同步 |
| `S3` | 方案或架构变更 | 已确认进入摄取或执行的完整方案、方案落实台账、跨模块重构、老项目原地换芯、高风险流程 | 完整 Engine 流程 |

## S0 轻量模式

S0 适用于不影响业务规则、数据、接口、权限、状态和模块边界的微小变更。

S0 必须做：

- 阅读 `AGENTS.md`。
- 查看相关代码或样式文件。
- 执行最小验证或说明无法验证原因。
- 最终说明不涉及记忆源写回、方案摄取、旧引擎清退和业务文档同步。

S0 不需要默认读取：

- 项目记忆源
- 方案台账或摄取记录
- 旧协作引擎文档
- 业务语义长文档
- 历史问题库

## 升级条件

S0 或 S1 任务出现以下任一情况，必须升级：

- 涉及业务规则、验收口径、字段语义或状态变化。
- 涉及项目风格画像、端侧交互习惯、命名偏好或工程约定变化。
- 涉及 API、数据库、权限、配置或自动化流程。
- 涉及模块职责、依赖方向或架构边界。
- 用户补充了可复用业务背景。
- 任务来源于已确认摄取、方案落实台账或执行中的完整方案。
- 发现文档、代码或测试存在冲突。
- 修复重复 Bug 或历史坑。
- 存在不可逆操作、数据迁移或高风险发布影响。

## 工作区模式门禁

修改前必须同时判断工作区模式。

| 模式 | 适用条件 | 必须记录 |
|---|---|---|
| 主工作区 | S0/S1、单任务、工作区干净、修改范围清晰、无共享文件冲突风险 | 说明未创建 Git worktree 的原因 |
| 独立 Git worktree | 多 Codex 任务并行、S2/S3、大改造、长期任务、实验性任务、主工作区不干净或共享文件冲突风险明显 | `worktree_path`、`worktree_branch`、合并状态和清理状态 |

使用独立 Git worktree 时，临时目录应放在项目目录外的统一位置。任务完成、合并或废弃后，必须清理临时工作树；存在未提交改动时不得自动删除。

## 任务状态门禁

S1 及以上任务建议维护运行态任务文件：

运行态任务文件可由当前协作引擎提供；薄入口项目不要求默认复制运行态文档。

S2 和 S3 必须维护运行态任务文件。

运行态任务文件用于记录：

- `change_level`
- `task_type`
- `docs_checked`
- `memory_source_writeback`
- `style_profile`
- `plan_intake`
- `plan_ledger`
- `old_engine_cleanup`
- `verification`
- `workspace_mode`
- `worktree_path`
- `worktree_branch`
- `worktree_cleanup`
- `strong_check`
- `git_commit`
- `open_questions`

## drift 检查

偏移检查应关注：

- 代码变更是否有任务状态或任务报告。
- S2/S3 是否评估项目记忆源写回和本地索引状态。
- 完整方案开发是否引用 `confirmed` 或 `active` 方案落实台账。
- 多任务并行或 S2/S3 变更是否记录工作区模式。
- 独立 Git worktree 是否登记路径、分支和清理状态。
- 代码、脚本、模板或协作文档修改是否记录强检查状态。
- 方案相关变更是否引用当前引擎的方案台账。
- 老项目升级是否完成旧引擎清退和项目记忆提取。
- 业务规则、字段、状态、接口或权限变化是否更新项目记忆源或对应文档。
- 项目风格、端侧习惯或命名约定变化是否更新项目记忆源。

薄入口项目不默认复制 `docs/agent/runtime/`。这类项目必须先通过 `scripts/check-agent.ps1` 校验入口、配置和项目记忆源，再由 `engine.source` 指向的当前协作引擎维护 S2/S3 运行态任务文件、任务报告和 drift 门禁。

## 门禁策略

| 等级 | 读取范围 | 文档同步 | 本地提交 | 报告要求 | 脚本检查 |
|---|---|---|---|---|---|
| `S0` | 入口 + 相关文件 | 说明不需要 | 修改文件后默认提交 | 极简说明 | 允许轻量通过 |
| `S1` | 入口 + 任务 workflow + 相关模块 | 评估是否需要 | 默认提交 | 标准摘要 | 检查任务状态建议 |
| `S2` | 增加项目记忆源、业务、模块文档 | 必须明确写回或说明不需要 | 默认提交并记录工作区模式 | 完整任务报告 | 检查任务状态必需 |
| `S3` | 增加方案台账、决策、历史问题、旧引擎清退记录 | 必须完整记录 | 默认提交并记录工作区模式 | 完整报告、影响矩阵和台账核对 | 强检查 |

## 禁止事项

- 不得用完整流程压迫 S0 微小变更。
- 不得把 S2/S3 伪装成 S0 来绕过文档同步。
- 不得只依赖 Agent 口头承诺遵守规则。
- 不得让代码变更在没有项目记忆源同步评估的情况下进入交付。
- 不得在方案讨论阶段修改项目记忆源或把未定版内容写成当前事实。
- 不得在没有 `confirmed` 或 `active` 方案落实台账时按零散对话直接实现完整方案。
- 不得让多个 Codex 任务并行共享同一个 working tree。
- 不得长期遗留未登记、未合并或未清理的临时 Git worktree。
- 不得把用户已有未提交改动混入本次本地提交。
- 不得在未获得用户明确授权时推送到远端仓库。
- 不得省略最终回复结尾的本地提交状态和“是否推送”。
