# 项目风格画像

## 目标

项目风格画像用于把不同项目、不同端侧在协作中形成的风格、习惯和实现偏好沉淀为项目级事实。

它解决的问题不是“统一所有项目的规范”，而是让 Agent 在进入某个项目时知道这个项目的前端、后端、客户端和其他端分别如何表达、组织和验证。

## 定位

项目风格画像是项目记忆源的一类，不是 Agent Context OS 的全局规范。薄入口项目推荐把这类事实写入 `.agent-context/memory-sources/`。

核心蓝图只规定记录方式：

- 哪些端侧需要有风格画像。
- 每条风格事实应标记状态、来源和适用范围。
- 任务触碰风格、交互、命名或工程习惯时应按需读取。
- 用户补充可复用风格偏好时应评估是否写回。

核心蓝图不规定具体答案：

- 不规定所有前端都使用同一种视觉风格。
- 不规定所有后端都使用同一种分层体系。
- 不规定所有客户端都使用同一种交互模型。
- 不把某个示例项目的做法提升为通用标准。

## 风格画像范围

| 端侧或层面 | 典型内容 | 推荐事实源 |
|---|---|---|
| 前端 | 视觉密度、组件复用、布局习惯、交互状态、文案语气、可访问性 | 项目记忆源、前端模块文档 |
| 后端 | 分层风格、命名习惯、错误处理、日志、接口组织、事务边界 | 项目记忆源、后端模块文档 |
| 客户端 | 平台交互、离线策略、性能体验、系统能力使用、发布约束 | 项目记忆源、客户端模块文档 |
| 自动化或脚本 | 命令风格、失败处理、重试、审计、人工介入点 | 项目记忆源、自动化模块文档 |
| 跨端体验 | 用户动作一致性、术语一致性、状态反馈、错误提示 | 项目记忆源、业务语义文档 |

## 事实状态

项目风格画像必须沿用项目记忆的事实状态：

| 状态 | 含义 | 使用规则 |
|---|---|---|
| `current` | 已确认的当前项目风格 | 可作为实现依据 |
| `draft` | 用户表达了方向，但细节未完整确认 | 使用前需要确认关键细节 |
| `assumption` | Agent 为推进任务做出的假设 | 不得沉淀为当前事实 |
| `deprecated` | 已废弃或被替代 | 只用于回溯 |

## 路由规则

出现以下情况时，Agent 应读取项目风格画像：

- UI、交互、文案、视觉密度、组件复用或可访问性变化。
- 后端分层、错误处理、日志、命名或接口组织变化。
- 客户端平台体验、离线行为、性能体验或发布约束变化。
- 用户补充“这个项目一贯怎么做”的偏好或习惯。
- 代码中出现多个相似实现，需要判断应复用哪种项目风格。
- 方案摄取中包含产品体验、工程体系或端侧实现习惯。

## 回写规则

Agent 发现以下内容时，应评估是否写回项目记忆源：

- 用户明确纠正了视觉、交互、文案、命名或工程习惯。
- 新功能确立了可复用的端侧模式。
- 重构沉淀出新的分层、命名或组织方式。
- 修复问题时发现现有风格规则缺失或互相冲突。
- 旧文档中的风格约定经过代码、测试或用户确认后仍然有效。

## 与其他文档的关系

| 文档 | 负责回答 |
|---|---|
| 项目记忆源 | 项目是什么、服务谁、各端如何表达、哪些习惯需要继承 |
| 当前业务文档 | 业务对象、流程、字段、状态和验收 |
| 当前架构文档 | 模块职责、依赖方向和禁止事项 |
| 当前技术文档 | 技术栈、运行环境和验证命令 |
| 模块文档 | 某个模块内部的具体实现边界和风险 |

## 禁止事项

- 不得把项目风格画像写成 Agent Context OS 全局规范。
- 不得把某个端侧的局部习惯强加到其他端侧。
- 不得把一次性视觉调整或临时实现写成 `current`。
- 不得在没有来源、状态和适用范围的情况下覆盖既有风格事实。
- 不得因为存在风格画像就跳过真实代码和现有实现搜索。
