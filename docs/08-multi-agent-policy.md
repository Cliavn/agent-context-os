# 多 Agent 策略

## 目标

多 Agent 策略用于在复杂任务中提升速度，同时控制成本和职责边界。

## 默认原则

多 Agent 是可选策略，不是默认行为。

默认使用单 Agent，只有在收益明显高于成本时才启用多 Agent。

用户要求按完整定稿方案开发时，如果方案落实台账包含多个任务、多个模块或多个可独立验证的验收点，默认启用多 Agent 协作；若不启用，主 Agent 必须说明任务很少、强串行或用户要求单 Agent 的原因。

## 适合启用的场景

| 场景 | 建议 |
|---|---|
| 跨前后端接口变更 | 可启用 |
| 大型重构 | 可启用 |
| 历史 Bug 链路排查 | 可启用 |
| 多业务模块并行分析 | 可启用 |
| confirmed / active 方案落实台账包含多个任务 | 默认启用 |
| 单文件小修 | 不启用 |
| 文案或样式小改 | 不启用 |

## 推荐角色

| 角色 | 职责 |
|---|---|
| 主 Agent | 决策、实施、整合最终结论 |
| 检索 Agent | 查找文档、代码入口和历史问题 |
| 审查 Agent | 检查风险、测试缺口和边界破坏 |
| 验证 Agent | 独立运行测试和复核验收结果 |

## 任务包原则

子 Agent 不应全仓自由探索。主 Agent 应提供明确任务包：

- 任务目标
- 方案落实台账路径和任务 ID
- 允许查看的目录或文档
- 输出格式
- 时间或 Token 预算
- 禁止修改的范围

## 汇总要求

主 Agent 必须汇总子 Agent 结果，去重、判断冲突，并对最终方案负责。

完整方案落地时，主 Agent 还必须按台账逐项回收：

- 每个任务 ID 是否完成。
- 验证或验收是否通过。
- 是否出现 `blocked`、`deferred` 或仍为 `pending` 的任务。
- 是否需要更新记忆、业务文档、模块文档、项目风格画像或 checklist。
