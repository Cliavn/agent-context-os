# 多 Agent 策略

## 目标

多 Agent 策略用于在复杂任务中提升速度，同时控制成本、职责边界和并行工作区隔离。

本文同时区分两类容易混淆的场景：

- 同一会话内由主 Agent 调度多个子 Agent。
- 多个 Codex 任务或会话同时指向同一个代码仓库。

前者重点是职责边界，后者重点是 working tree（工作区）隔离。

## 默认原则

多 Agent 是可选策略，不是默认行为。

默认使用单 Agent，只有在收益明显高于成本时才启用多 Agent。

用户要求按完整定稿方案开发时，如果方案落实台账包含多个任务、多个模块或多个可独立验证的验收点，默认启用多 Agent 协作；若不启用，主 Agent 必须说明任务很少、强串行或用户要求单 Agent 的原因。

默认不为所有改动创建 Git worktree（工作树）。S0/S1 单任务小改动可以在主工作区完成；只要存在多个 Codex 任务并行、大改造、长期任务、实验性任务、共享文件冲突风险或 S2/S3 变更，就必须使用独立 Git worktree 和独立 branch（分支）。

## 适合启用的场景

| 场景 | 建议 |
|---|---|
| 跨前后端接口变更 | 可启用 |
| 大型重构 | 可启用 |
| 历史 Bug 链路排查 | 可启用 |
| 多业务模块并行分析 | 可启用 |
| confirmed / active 方案落实台账包含多个任务 | 默认启用 |
| 单文件小修 | 不启用 |
| 文案或样式小改 | 不启用 |

## 多 Codex 任务并行开发

多个 Codex 任务或会话并行开发同一仓库时，不应共享同一个项目目录。

| 场景 | 工作区模式 |
|---|---|
| S0/S1 小改动、单任务、工作区干净、修改范围清晰 | 可直接使用主工作区 |
| 多个 Codex 任务同时运行同一仓库 | 必须使用独立 Git worktree |
| S2/S3、大改造、跨文件或跨模块变更 | 必须使用独立 Git worktree |
| 任务持续时间较长、需要反复试验或可能废弃 | 建议使用独立 Git worktree |
| 当前主工作区已有未提交改动，无法安全区分来源 | 必须使用独立 Git worktree |
| 多个任务可能修改同一共享文件 | 必须隔离开发，并在集成阶段处理冲突 |

独立工作树必须满足：

- 每个任务一个 Git worktree 和一个 branch。
- 临时工作树放在项目目录外的统一位置，不散落在项目根目录内。
- 任务运行态记录 `workspace_mode`、`worktree_path`、`worktree_branch` 和 `worktree_cleanup`。
- 任务完成后只提交本任务改动，集成阶段再合并分支。
- 分支已合并或任务废弃后，必须执行 `git worktree remove <worktree路径>` 和 `git worktree prune`。
- 若工作树仍有未提交改动，不得自动删除；必须先确认提交、丢弃或保留现场。

## 工作树生命周期

```text
判断变更等级和并行风险
> 选择主工作区或独立 Git worktree
> 创建独立 branch
> 在 current-task 中登记路径、分支和清理状态
> 完成修改、验证和本地提交
> 集成分支合并
> 删除临时 worktree
> prune 清理失效记录
> 删除已合并临时分支
```

## 推荐角色

| 角色 | 职责 |
|---|---|
| 主 Agent | 决策、实施、整合最终结论 |
| 检索 Agent | 查找文档、代码入口和历史问题 |
| 审查 Agent | 检查风险、测试缺口和边界破坏 |
| 验证 Agent | 独立运行测试和复核验收结果 |

## 任务包原则

子 Agent 不应全仓自由探索。主 Agent 应提供明确任务包：

- 任务目标
- 方案落实台账路径和任务 ID
- 允许查看的目录或文档
- 输出格式
- 时间或 Token 预算
- 禁止修改的范围

## 汇总要求

主 Agent 必须汇总子 Agent 结果，去重、判断冲突，并对最终方案负责。

完整方案落地时，主 Agent 还必须按台账逐项回收：

- 每个任务 ID 是否完成。
- 验证或验收是否通过。
- 是否出现 `blocked`、`deferred` 或仍为 `pending` 的任务。
- 是否需要更新记忆、业务文档、模块文档、项目风格画像或 checklist。
- 若使用独立 Git worktree，路径、分支、合并状态和清理状态是否已回收。
