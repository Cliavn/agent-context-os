# 方案理解与摄取系统

## 目标

方案理解与摄取系统用于处理用户输入的完整业务方案、产品方案、技术方案或执行计划。

它的目标不是保存方案全文，而是先理解方案初衷和业务意义，再拆解实现逻辑、系统影响和待确认问题，最后把有效内容融合进项目记忆源、业务文档、模块文档、决策记录和任务结构。

## 核心原则

1. **先判断阶段**：用户只是在讨论、比较或推敲方案时，不得修改项目记忆源或当前事实。
2. **确认后再摄取**：只有用户明确要求定版、执行、记录草案或写回时，才进入方案摄取、方案落实台账或文档回写。
3. **先理解，后拆解**：不得在未理解方案初衷和业务意义前直接拆任务。
4. **先分层，后入库**：方案内容必须按层级进入不同事实源，不得只保存为单一长文本。
5. **保留来源**：每条迁移出的事实、规则或任务都应能追溯到原方案。
6. **状态明确**：区分已确认事实、方案建议、假设、开放问题和冲突。
7. **按需路由**：后续任务优先读取方案索引和结构化摘要，不默认读取方案全文。
8. **冲突可见**：方案与现有项目记忆源、代码或测试冲突时必须报告。

## 方案五层模型

| 层级 | 名称 | 要回答的问题 | 输出位置 |
|---|---|---|---|
| 1 | 方案意图层 | 用户为什么提出这个方案，要解决什么痛点 | 方案摄取记录、项目记忆源 |
| 2 | 业务意义层 | 涉及哪些用户、对象、流程、规则和验收 | 业务文档、项目记忆源 |
| 3 | 实现逻辑层 | 需要哪些能力、模块、接口、数据和权限变化 | 实现说明、模块文档 |
| 4 | 系统影响层 | 对已有功能、边界、风险、测试和发布有什么影响 | 影响矩阵、决策记录 |
| 5 | 融合更新层 | 哪些内容写回当前事实，哪些保留为开放问题 | memory、modules、business、reports |

## 方案状态

| 状态 | 含义 | 使用规则 |
|---|---|---|
| `current` | 已确认并作为当前事实 | 可作为实现依据 |
| `proposed` | 方案建议，尚未落地或确认 | 不得覆盖当前事实 |
| `assumption` | Agent 推测出的内容 | 必须等待用户确认 |
| `open` | 需要用户或负责人确认 | 进入开放问题 |
| `conflict` | 与现有事实、代码或测试冲突 | 必须报告并处理 |
| `deprecated` | 已被新方案替代 | 只用于回溯 |

## 方案摄取流程

1. 判断当前是讨论阶段（`discussion_only`）、草案记录（`draft_record`）、定版摄取还是执行落地。
2. 如果仍处于讨论阶段，只在对话中输出候选方案、取舍、风险和待确认问题，不修改协作文档。
3. 获得用户对定版、执行、记录草案或写回的明确确认后，识别方案类型和来源。
4. 提取方案初衷、业务目标、非目标和成功标准。
5. 拆解业务对象、用户角色、业务流程、字段规则、状态变化和验收口径。
6. 分析实现逻辑、模块影响、接口影响、数据影响、权限影响和运行态影响。
7. 生成系统影响矩阵。
8. 检查与当前项目记忆源、业务文档、模块文档和代码事实是否冲突。
9. 将内容标记为 `current`、`proposed`、`assumption`、`open` 或 `conflict`。
10. 如果用户要求形成开发顺序、任务清单、记录草稿或准备落地，按当前协作引擎创建或更新方案落实台账，状态为 `draft`。
11. 按确认范围写入对应上下文；记录草案时只能写入 `proposed`、`open` 或 `assumption`，不得覆盖 `current`。
12. 生成方案摄取报告。
13. 后续任务按方案索引和方案落实台账读取相关片段，不默认读取方案全文。

## 方案定版门禁

以下情况视为讨论阶段：用户说“讨论一下”、“帮我看看方案”、“有没有更好方案”、“先聊思路”，或没有明确要求写入文档、更新上下文、定版、执行、开始实现。

以下情况才视为进入摄取或写回：用户明确表示“定版”、“按这个执行”、“开始实现”、“写入文档”、“更新上下文”、“记录到 intake.md”或“记录草案”。

以下情况才视为进入方案落实台账：用户明确表示“形成开发顺序”、“列任务清单”、“整理成开发计划”、“记录草稿”、“准备落地”或“按完整方案开发”。

讨论阶段允许 Agent 输出：

- 方案理解摘要。
- 候选路径和取舍。
- 风险、冲突和待确认问题。
- 建议的后续确认清单。

讨论阶段禁止 Agent 修改：

- 项目记忆源。
- 业务文档、模块文档、决策记录和任务报告。
- 当前协作引擎的方案台账或方案索引。

## 方案落实台账

当用户要求形成开发顺序或任务清单时，Agent 应把讨论结论整理为方案落实台账。台账不是原始会话记录，而是 `draft` 状态的结构化开发清单。

台账必须至少包含：

- 方案目标和非目标。
- 已确认点和待确认点。
- 开发顺序。
- 任务 ID、来源摘要、依赖、验收口径和状态。
- 多 Agent 分工和回收结果。
- 完成核对和变更记录。

`draft` 台账不得作为实现依据。用户确认后，台账状态才能转为 `confirmed`；用户要求开始完整开发时，台账转为 `active`，并按顺序逐项执行。

## 系统影响矩阵

方案摄取必须评估以下影响：

| 影响类型 | 检查问题 |
|---|---|
| 用户影响 | 哪些用户角色、权限或操作路径变化 |
| 业务影响 | 哪些对象、流程、字段、状态和异常变化 |
| 模块影响 | 哪些模块职责、边界或依赖方向变化 |
| 接口影响 | 哪些 API、事件、命令或集成契约变化 |
| 数据影响 | 哪些数据结构、迁移、兼容或回滚风险变化 |
| 项目风格影响 | 哪些前端、后端、客户端或其他端侧风格和工程习惯变化 |
| UI 影响 | 哪些页面、交互状态、文案或可访问性变化 |
| 自动化影响 | 哪些任务编排、重试、日志或审计变化 |
| 测试影响 | 哪些单元、集成、回归或手工验收需要补充 |
| 发布影响 | 是否需要配置、开关、灰度、回滚或监控 |

## 融合目标

方案内容应按类型写入不同位置：

| 内容 | 推荐位置 |
|---|---|
| 普通讨论摘要 | 仅当前对话，不写入项目记忆源 |
| 方案摘要和来源 | 当前协作引擎的方案摄取记录 |
| 开发顺序和任务清单 | 当前协作引擎的方案落实台账 |
| 业务背景和确认事实 | 项目记忆源 |
| 用户角色、流程、字段、状态 | 业务语义文档 |
| 模块职责和边界变化 | 模块文档 |
| 项目风格、交互习惯和端侧工程约定 | 项目记忆源 |
| 技术取舍和重大决策 | 决策记录 |
| 实施计划和任务拆分 | 实现说明或任务报告 |
| 未确认问题 | 项目记忆源中的开放问题 |
| 与旧协作引擎关系 | 旧引擎清退记录 |

## 禁止事项

- 不得只把方案全文保存为单文件后结束。
- 不得未理解业务初衷就直接拆任务。
- 不得在方案讨论阶段修改项目记忆源或当前事实。
- 不得把方案建议直接覆盖当前事实。
- 不得把 Agent 推测标记为 `current`。
- 不得忽略方案对已有功能、接口、数据、权限和测试的连带影响。
- 不得让后续任务默认读取方案全文。
- 不得在没有 `confirmed` 或 `active` 台账时按零散对话直接实现完整方案。
