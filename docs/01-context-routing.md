# 上下文路由

## 目标

上下文路由的目标是让 Agent 用最少 Token 读取最有用的信息，避免每次全仓扫描或全量阅读长文档。

## 分层加载策略

Agent 应按以下顺序加载上下文：

| 层级 | 内容 | 读取条件 |
|---|---|---|
| L0 | `AGENTS.md` | 每次任务必读 |
| L1 | `docs/agent/00-index.md` | 每次任务必读 |
| L2 | 接入状态 | 首次接入、老项目迁移或上下文缺失时读取 |
| L3 | 方案摄取索引 | 用户输入完整方案、计划或方案文档时读取 |
| L4 | 变更分级 | 修改前判断 S0-S3，决定后续读取范围 |
| L5 | 任务 workflow 和 checklist | 按任务类型读取 |
| L6 | 项目记忆 | 涉及业务背景、用户补充、开放问题或历史理解时，优先检索 `docs/agent/memory-store/`，再按命中结果读取 `memory.md` 或事实源 |
| L7 | 项目风格画像 | 涉及端侧风格、交互习惯、命名偏好或工程体系时读取 |
| L8 | 相关模块卡片 | 修改模块前读取 |
| L9 | 业务语义摘要 | 涉及业务规则时读取 |
| L10 | 历史问题和长文档 | 复杂 bug、冲突或回归风险时读取 |

## 任务路由示例

| 任务类型 | 必读上下文 |
|---|---|
| 渐进式接入 | progressive-adoption workflow、adoption checklist、旧文档索引 |
| 方案理解与摄取 | plan-intake workflow、plan-intake checklist、intake、memory、相关业务和模块文档 |
| S0 微小变更 | AGENTS、相关代码或样式文件、最小验证 |
| 修复 Bug | bug-fix workflow、bug checklist、相关模块卡片、历史问题库 |
| 新增功能 | new-feature workflow、项目记忆、模块卡片、业务规则、验收口径 |
| 重构 | refactor workflow、架构边界、模块依赖、现有测试 |
| 字段规则变更 | 项目记忆、字段规则文档、业务流程、状态机、回归用例 |
| UI 改造 | UI workflow、项目风格画像、相关页面模块卡片 |
| 端侧风格或工程习惯调整 | 项目风格画像、相关模块卡片、必要的业务或架构文档 |
| 数据库变更 | storage 模块卡片、migration 规则、回滚方案 |
| 发布或提交 | version-control workflow、发布 checklist |

## 过期文档处理

文档应包含元信息：

```md
状态：current / supplement / history / deprecated
最后校验日期：<YYYY-MM-DD>
关联代码路径：
关联测试：
适用范围：
过期风险：
```

规则：

- `current` 可作为主上下文。
- `history` 只用于回溯，不作为当前实现依据。
- 超过项目约定期限未校验的文档，必须对照代码确认。
- 文档和代码冲突时，以代码和测试为准，并报告文档过期。

## Token 控制原则

- 先检索记忆摘要，不读全文。
- 有 `docs/agent/memory-store/` 时，先按任务类型、模块、文件路径、接口名或业务对象召回相关记忆。
- 检索命中后，只深读与当前任务相关的事实源、模块文档或业务文档。
- 先读 current，不读 history。
- 先读模块卡片，再读长文档。
- 老项目首次接入只建立旧文档索引，不读取全文。
- 完整方案先读结构化摄取记录，不默认读取方案全文。
- S0 微小变更不默认读取 memory、memory-store、intake、legacy 和业务长文档。
- 只有出现冲突、复杂 bug 或高风险变更时才深读。
- 多 Agent 必须有明确触发条件和任务包。

## 上下文回写原则

- 用户补充的业务背景、验收口径、字段含义、状态规则或历史坑，必须评估是否写回。
- 用户补充的前端、后端、客户端或其他端侧风格习惯，必须评估是否写回项目风格画像。
- 方案仍处于讨论、比较或推敲阶段时，不触发协作文档写回；除非用户明确要求记录草案、定版、执行或写回。
- 已确认事实写入对应当前文档；未确认信息写入开放问题或标记为 `draft`。
- Agent 推测只能标记为 `assumption`，不得覆盖 `current`。
- 每次任务报告必须说明本次是否需要记忆回写。
- 启用检索式项目记忆时，任务报告还必须说明是否新增、更新、废弃或校验了 `docs/agent/memory-store/` 中的记忆。

## 渐进式接入原则

- 首次接入老项目时，只读取入口、接入 workflow、接入 checklist 和必要目录结构。
- 旧文档只建立路径和主题索引，不默认总结全文。
- 旧文档内容必须经过代码、测试或用户确认后，才能迁移为当前事实。
- 接入完成后，后续任务再按需补齐模块、业务和历史问题上下文。

## 方案摄取原则

- 方案进入项目时，必须先理解初衷、业务意义、实现逻辑和系统影响。
- 未定版方案只在对话中讨论，不得自动进入 `intake.md`、`memory.md`、业务文档或模块文档。
- 方案全文只作为来源，不作为每次任务默认上下文。
- 方案建议必须标记为 `proposed`，不得直接覆盖 `current`。
- 与当前事实冲突的方案内容必须标记为 `conflict`。
- 后续任务优先读取 `intake.md` 中的索引、状态和影响范围。

## 变更分级原则

- 修改前先判断 `S0`、`S1`、`S2` 或 `S3`。
- `S0` 只读取入口规则和相关文件，并说明不需要文档同步。
- `S1` 读取任务 workflow 和相关模块上下文。
- `S2` 必须评估项目记忆、项目风格画像、业务文档、模块文档和验证影响。
- `S3` 必须评估方案摄取、旧文档、决策记录、历史问题和系统影响矩阵。
