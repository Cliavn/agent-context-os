# 业务建模

## 目标

业务建模用于把任意项目的业务知识拆成 Agent 可读取、可继承、可验证的结构。

通用的不是业务内容，而是描述业务的结构。

## 六类业务语义

每个业务模块至少应描述以下六类信息：

| 类型 | 说明 |
|---|---|
| 业务对象 | 用户、订单、任务、资产、账号、文件等核心对象 |
| 业务流程 | 主流程、分支流程、异常流程 |
| 字段规则 | 字段含义、来源、映射、默认值、校验 |
| 状态流转 | 状态定义、允许流转、禁止流转 |
| 异常场景 | 边界条件、失败路径、人工处理点 |
| 验收口径 | 如何判断业务正确、测试数据和人工检查点 |

## 推荐文档拆分

每个业务模块可以使用以下结构：

```text
business/<模块名>/
├─ module-overview.md
├─ business-flow.md
├─ field-rules.md
├─ state-machine.md
├─ api-contract.md
├─ edge-cases.md
└─ acceptance.md
```

## 建模要求

- 业务规则必须能被代码实现和测试引用。
- 字段规则必须说明来源、含义、默认值、校验和空值处理。
- 状态流转必须写清允许和禁止的转换。
- 异常场景必须说明用户看到什么、系统记录什么、是否可重试。
- 验收口径必须能变成测试、手工检查或回归清单。

## 反模式

- 把所有业务规则写进一个超长文档。
- 只写页面文案，不写业务对象和状态。
- 只记录成功路径，不记录失败路径。
- 历史方案和当前规则混写。
- 字段名、接口名和展示名没有映射关系。
