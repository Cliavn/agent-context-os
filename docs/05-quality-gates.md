# 质量门禁

## 目标

质量门禁用于定义任务完成的最低标准，提升代码一次通过率。

## 完成定义

一次任务只有满足以下条件，才算完成：

- 修改范围与需求一致。
- 已遵守模块边界。
- 已执行相关测试或说明无法执行的原因。
- 已检查错误路径和边界条件。
- 已评估项目记忆源是否需要同步更新。
- 已评估本地检索索引是否需要重建或增量更新。
- 来源于完整方案落地时，已按方案落实台账逐项核对。
- 已判断变更等级，并按等级选择轻量模式或完整门禁。
- 已评估运行态是否需要重启。
- 代码或协作文档修改完成并通过必要验证后，已创建本地 Git 提交；如未提交，必须说明原因。
- 已运行强检查矩阵；如有检查不适用或无法运行，必须说明原因、替代验证和剩余风险。
- 最终回复结尾已包含本地提交状态和“是否推送”。
- 已在任务报告中说明风险和剩余问题。

## 验证矩阵

| 变更类型 | 推荐验证 |
|---|---|
| S0 微小变更 | 最小手工检查或相关局部验证 |
| 纯文档 | 结构检查、链接检查 |
| 老项目原地换芯 | 旧引擎清退检查、项目记忆源提取检查、极薄入口检查 |
| 方案理解与摄取 | 方案意图检查、业务意义检查、系统影响矩阵、冲突和开放问题检查 |
| 方案落实台账 | 开发顺序检查、任务完整性检查、验收口径检查、任务状态检查 |
| 项目风格画像 | 端侧适用范围检查、现有实现一致性检查、状态和来源检查 |
| 前端 UI | 类型检查、构建、关键交互手工验收 |
| 后端逻辑 | 单元测试、集成测试、接口验证 |
| 数据库 | migration 测试、回滚检查、数据兼容验证 |
| 自动化流程 | dry-run、失败路径、重试和取消验证 |
| Bug 修复 | 复现证据、回归测试、防复发规则 |
| 重构 | 行为保持测试、接口兼容检查 |

## 强检查矩阵

代码、脚本、模板或协作文档修改后，必须优先运行：

```powershell
scripts/check-agent-strong.ps1
```

强检查至少覆盖：

- `git diff --check`：检查未暂存差异中的空白错误。
- `git diff --cached --check`：检查已暂存差异中的空白错误。
- `scripts/check-agent-context-os.ps1`：检查蓝图结构、必需文件、模板同步和核心引用。
- `scripts/check-agent-project.ps1`：检查极薄项目入口、项目记忆源和本地索引排除规则。
- `scripts/check-agent-worktrees.ps1`：检查 Git worktree（工作树）是否嵌套在仓库内、是否存在可清理残留。
- `scripts/check-agent-drift.ps1`：检查代码类变更是否留下任务状态和文档同步痕迹。

如果某个检查不适用于当前项目，最终报告必须说明跳过原因；如果检查失败，不得宣称交付完成，除非用户明确接受失败状态和剩余风险。

## 代码通过率定义

建议把代码通过率定义为：

```text
通过率 = 首次交付后无需返工即可通过约定验证的任务数 / 总任务数
```

可按项目细化为：

- 自动化测试通过率
- 构建通过率
- 人工验收通过率
- 回归缺陷率
- 同类 Bug 复发率

## 项目记忆源同步条件

以下情况必须同步更新项目记忆源，或说明为什么无需写回：

- 新增或删除模块。
- 改变模块职责或依赖方向。
- 改变项目风格、端侧交互习惯、命名偏好或工程约定。
- 改变公共接口、字段、状态或配置。
- 引入新技术栈或关键依赖。
- 修复重复 Bug 或发现规则缺口。
- 改变验证命令、启动方式或发布流程。
- 用户补充了可复用业务背景、验收口径、历史坑或开放问题。
- 老项目升级时从旧协作引擎中提取了有效项目事实。
- 用户明确确认完整方案需要定版、执行或写回，或已确认方案影响业务规则、模块边界、接口、数据、权限和验收标准。
- 用户要求形成开发顺序、任务清单、记录草稿、定稿或按完整方案开发。

## 记忆源回写门禁

任务结束前必须说明：

- 本次是否产生新的项目事实、业务事实或用户补充背景。
- 本次是否产生新的项目风格事实或端侧约定。
- 已写回到哪些项目记忆源，或为什么不需要写回。
- 本地检索索引是否需要重建或增量更新。
- 是否存在不能安全假设、需要用户确认的开放问题。

## 方案讨论门禁

用户仍在讨论、比较或推敲方案时，Agent 必须说明当前仍是讨论阶段，并且不得修改项目记忆源、业务文档、模块文档、决策记录或项目风格画像。

以下信号才允许进入方案摄取或文档写回：用户明确表示“定版”、“按这个执行”、“开始实现”、“写入文档”、“更新上下文”或“记录草案”。记录草案时只能写入 `proposed`、`open` 或 `assumption`，不得覆盖 `current`。

## 方案落实台账门禁

当用户要求形成开发顺序、任务清单、记录草稿或准备落地时，Agent 必须创建或更新 `draft` 台账，而不是保存原始会话。

当用户要求按完整方案开发时，Agent 必须先确认存在 `confirmed` 或 `active` 台账。没有定稿台账时，不得直接按零散对话实现；应先整理台账并请求用户确认。

完整方案交付前必须说明：

- 台账路径和状态。
- 任务总数、完成、阻塞、延期和未开始数量。
- 是否存在没有原因的 `pending` 任务。
- 哪些任务触发了项目记忆源、业务文档、模块文档或项目风格画像写回。

## 方案摄取门禁

完整方案进入项目后，必须说明：

- 是否已经理解方案初衷、业务意义和非目标。
- 是否已经生成系统影响矩阵。
- 哪些内容是 `current`、`proposed`、`assumption`、`open` 或 `conflict`。
- 哪些内容写回了项目记忆源、业务文档、模块文档或决策记录。

## 执行门禁

- S0 微小变更可以走轻量模式，但必须说明不涉及业务规则、记忆源写回、方案摄取和旧引擎清退。
- S2/S3 必须维护或更新任务状态、记忆源写回结论和本地索引状态。
- 代码修改后应运行 drift 或项目检查，防止项目记忆源和真实实现偏移。
- 代码、脚本、模板或协作文档修改后应运行强检查矩阵；优先使用 `scripts/check-agent-strong.ps1`。
- 代码或协作文档修改后，默认执行本地 Git 提交；提交前必须完成必要验证并只暂存本次相关文件。

## 本地提交门禁

任务完成时默认创建本地 Git 提交，除非出现以下情况：

- 用户明确要求本次不要提交。
- 任务没有产生文件变更。
- 验证失败且用户没有明确接受失败状态。
- 工作区存在无法安全区分来源的未提交改动。
- 当前环境不是 Git 仓库，或仓库状态阻止提交。

本地提交规则：

- 提交前必须检查工作区状态。
- 只暂存本次 Agent 相关文件。
- 提交信息必须能说明需求或修复内容。
- 不得提交账号、密钥、token、cookie 或真实隐私数据。
- 不得自动推送到远端仓库；推送必须获得用户明确授权。
- 最终回复结尾必须写明本地提交状态，并保留“是否推送”字样。
