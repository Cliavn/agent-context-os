# 质量门禁

## 目标

质量门禁用于定义任务完成的最低标准，提升代码一次通过率。

## 完成定义

一次任务只有满足以下条件，才算完成：

- 修改范围与需求一致。
- 已遵守模块边界。
- 已执行相关测试或说明无法执行的原因。
- 已检查错误路径和边界条件。
- 已评估文档是否需要同步更新。
- 已评估运行态是否需要重启。
- 已在任务报告中说明风险和剩余问题。

## 验证矩阵

| 变更类型 | 推荐验证 |
|---|---|
| 纯文档 | 结构检查、链接检查 |
| 前端 UI | 类型检查、构建、关键交互手工验收 |
| 后端逻辑 | 单元测试、集成测试、接口验证 |
| 数据库 | migration 测试、回滚检查、数据兼容验证 |
| 自动化流程 | dry-run、失败路径、重试和取消验证 |
| Bug 修复 | 复现证据、回归测试、防复发规则 |
| 重构 | 行为保持测试、接口兼容检查 |

## 代码通过率定义

建议把代码通过率定义为：

```text
通过率 = 首次交付后无需返工即可通过约定验证的任务数 / 总任务数
```

可按项目细化为：

- 自动化测试通过率
- 构建通过率
- 人工验收通过率
- 回归缺陷率
- 同类 Bug 复发率

## 文档同步条件

以下情况必须同步更新项目协作文档：

- 新增或删除模块。
- 改变模块职责或依赖方向。
- 改变公共接口、字段、状态或配置。
- 引入新技术栈或关键依赖。
- 修复重复 Bug 或发现规则缺口。
- 改变验证命令、启动方式或发布流程。
