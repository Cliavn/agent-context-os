# 系统总览

## 目标

Agent Context OS 是一套面向 AI Agent 协助开发的通用协作引擎蓝图。它的目标不是把大量协作文档复制进每个用户项目，而是让协作引擎、项目记忆源和本地检索索引各司其职。

核心目标：

- Agent 进入项目时有稳定入口和清晰门禁。
- 用户项目保持极薄项目入口，不堆积协作引擎副本。
- 项目业务背景、交互习惯、开发习惯和历史坑沉淀为可审查项目记忆源。
- 本地检索索引用于快速召回相关摘要和路径，减少 Token 消耗。
- 老项目升级时原地换芯，不长期保留两套协作结构。

## 解决的问题

很多项目在使用 Agent 协助开发时会遇到以下问题：

1. Agent 每次都重新猜项目结构，重复消耗 Token。
2. 协作规则和项目业务事实混在一起，升级引擎时容易拖泥带水。
3. 业务规则散落在对话、历史文档和代码里，无法继承。
4. 文档越来越多，Agent 读取成本越来越高。
5. 向量库文件如果直接进入 Git，会造成仓库膨胀和团队同步问题。
6. 架构边界不清，Agent 容易跨模块乱改。
7. Bug 修复只停留在代码层，没有沉淀防复发规则。
8. 多 Agent 能提速，但没有成本和职责边界，容易变贵。
9. 文档过期后仍被 Agent 误用，导致上下文错误。
10. 老项目升级协作引擎时，被迫保留旧兼容层，项目结构变脏。
11. 完整方案被保存为单一长文本，后续 Agent 漏读或误解业务初衷。
12. 多个 Codex 任务并行指向同一项目目录时，共享同一个 working tree（工作区）和 Git index（暂存区），容易互相污染改动。

## 核心模型

Agent Context OS 按三层运行：

| 层 | 名称 | 作用 |
|---|---|---|
| 1 | 协作引擎层 | 管理 Agent 如何工作、如何拆任务、如何判断 S0-S3、如何验证和提交 |
| 2 | 项目记忆源层 | 管理用户项目的业务事实、开发习惯、交互习惯、实现约束和历史坑 |
| 3 | 本地检索索引层 | 从项目记忆源和代码证据生成向量、全文和关键词索引，用于快速召回 |

协作引擎层保留在 Agent Context OS 本体或版本化引擎包中。用户项目默认只保留：

```text
AGENTS.md
.agent-context/config.json
.agent-context/memory-sources/
scripts/check-agent.ps1
```

## 协作引擎能力

协作引擎统一管理：

- 入口规则和上下文路由。
- S0-S3 变更等级和执行门禁。
- 方案理解、方案摄取和方案落实台账。
- 质量门禁、验证矩阵和任务报告。
- 单 Agent / 多 Agent 协作策略。
- Git worktree（工作树）并行开发隔离。
- 项目记忆源写回和本地索引重建规则。

这些规则不应作为完整 `docs/agent/` 副本进入每个用户项目。

## 项目记忆边界

项目记忆包括：

- 项目目标、非目标、用户和阶段。
- 业务对象、流程、字段、状态、异常和验收口径。
- 用户使用交互习惯、端侧风格、团队开发习惯。
- 模块职责、依赖方向、实现约束和验证命令。
- 历史问题、根因、防复发规则和证据路径。

项目记忆不包括协作引擎的通用 workflow、checklist、门禁规则或任务拆解方式。

## 事实优先级

当记忆、索引、文档、代码、测试和对话存在冲突时，默认按以下顺序判断：

```text
真实代码 / 测试结果
> 用户确认
> 项目记忆源
> 决策记录
> 本地检索索引
> 历史文档
> 对话记录
> Agent 推测
```

本地检索索引只负责定位上下文，不作为最终事实源。

## 老项目升级

老项目升级不是增加兼容层，而是原地换芯：

1. 识别旧协作引擎规则和其中混入的项目记忆。
2. 将有效项目记忆提取到 `.agent-context/memory-sources/`。
3. 删除或清退旧协作引擎文件。
4. 写入极薄项目入口和 `.agent-context/config.json`。
5. 重建本地检索索引。
6. 按新版协作引擎门禁执行后续任务。

升级后用户项目只保留一套当前结构。

## 通用性边界

核心蓝图只定义协作引擎如何运作、项目记忆如何组织、本地索引如何使用，不内置任何具体项目的业务答案。

示例可以出现具体项目形态，但必须放在 `examples/` 下，并明确标注为示例。
