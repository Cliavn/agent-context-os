# 系统总览

## 目标

Agent Context OS 是一套面向 AI Agent 协助开发的通用蓝图。它帮助项目以渐进方式接入协作引擎，并在开发过程中建立清晰的上下文、业务语义、项目风格、架构边界、质量流程和成本控制规则。

## 解决的问题

很多项目在使用 Agent 协助开发时会遇到以下问题：

1. Agent 每次都重新猜项目结构，重复消耗 Token。
2. 业务规则散落在对话、历史文档和代码里，无法继承。
3. 架构边界不清，Agent 容易跨模块乱改。
4. Bug 修复只停留在代码层，没有沉淀防复发规则。
5. 多 Agent 能提速，但没有成本和职责边界，容易变贵。
6. 文档过期后仍被 Agent 误用，导致上下文错误。
7. 老项目首次接入时被迫全量整理旧文档，接入成本过高。
8. 完整方案被保存为单一长文本，后续 Agent 漏读或误解业务初衷。
9. 微小改动如果也走完整流程，会导致 Token 和时间成本过高。
10. 项目的前端、后端、客户端等端侧风格和工程习惯没有沉淀，Agent 容易把一个项目的偏好误当成所有项目的规范。

## 核心模型

Agent Context OS 将项目协作拆成十五层：

| 层级 | 名称 | 作用 |
|---|---|---|
| 1 | 入口层 | 规定 Agent 如何开始工作 |
| 2 | 接入层 | 支持新项目完整接入和老项目渐进式接入 |
| 3 | 路由层 | 按任务类型决定读取哪些上下文 |
| 4 | 项目画像层 | 描述项目目标、非目标、用户和阶段 |
| 5 | 架构边界层 | 描述模块职责、依赖方向和禁止事项 |
| 6 | 业务语义层 | 描述业务对象、流程、字段、状态和异常 |
| 7 | 方案摄取层 | 理解完整方案的初衷、业务意义、实现逻辑和系统影响 |
| 8 | 执行门禁层 | 按 S0-S3 判断变更等级，控制读取范围和检查强度 |
| 9 | 项目记忆层 | 沉淀用户补充背景、项目事实、开放问题和记忆回写规则 |
| 10 | 检索记忆层 | 用结构化记录、全文检索和可选向量检索快速召回项目记忆 |
| 11 | 项目风格层 | 沉淀前端、后端、客户端等端侧的风格、交互和工程习惯 |
| 12 | 实现说明层 | 在开发前明确目标、方案、影响和验证 |
| 13 | 质量门禁层 | 用 workflow、checklist 和测试控制交付质量 |
| 14 | 历史问题层 | 沉淀问题根因、防复发规则和回归测试 |
| 15 | 成本控制层 | 控制 Token、上下文加载和多 Agent 触发 |

## 事实优先级

当文档、代码、测试和对话存在冲突时，默认按以下顺序判断：

```text
真实代码 / 测试结果
> 当前有效文档
> 决策记录
> 检索记忆
> 历史文档
> 对话记录
> Agent 推测
```

Agent 发现文档与代码不一致时，不应直接忽略冲突，而应在任务报告中说明并建议更新文档。

用户在协作中补充的可复用业务背景、项目事实或验收口径，不应只停留在当前对话。Agent 应判断是否写回项目记忆、检索记忆、业务文档、模块文档、历史问题库或 checklist。

老项目首次接入时，旧文档默认不作为当前事实。Agent 应先建立旧文档索引，再在具体任务中按需读取、验证和迁移。

用户输入完整方案时，方案全文不应成为默认上下文。Agent 应先理解方案初衷和业务意义，再拆解系统影响，并把有效内容写入对应事实源。

如果用户只是在讨论、比较或推敲方案，且没有明确要求定版、执行、记录草案或写回上下文，Agent 不得修改协作文档，也不得把讨论内容写成 `current` 或 `proposed`。此时只能在对话中整理候选方案、风险、取舍和待确认问题；只有获得明确确认后，才能进入方案摄取或记忆回写流程。

微小变更不应承担完整流程成本。Agent 应先判断变更等级，S0 走轻量模式，S2/S3 才强制完整记忆、文档和方案同步评估。

## 通用性边界

核心蓝图只定义“如何描述项目和业务”，不内置任何具体项目的业务答案。

核心蓝图只定义“如何沉淀项目风格和端侧习惯”，不内置任何统一视觉、交互、后端分层或客户端实现规范。前端、后端、客户端等端侧风格应写入项目自己的事实源，而不是写成 Agent Context OS 的全局答案。

示例可以出现具体项目形态，但必须放在 `examples/` 下，并明确标注为示例。
