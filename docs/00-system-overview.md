# 系统总览

## 目标

Agent Context OS 是一套面向 AI Agent 协助开发的通用蓝图。它帮助项目在开发前建立清晰的上下文、业务语义、架构边界、质量流程和成本控制规则。

## 解决的问题

很多项目在使用 Agent 协助开发时会遇到以下问题：

1. Agent 每次都重新猜项目结构，重复消耗 Token。
2. 业务规则散落在对话、历史文档和代码里，无法继承。
3. 架构边界不清，Agent 容易跨模块乱改。
4. Bug 修复只停留在代码层，没有沉淀防复发规则。
5. 多 Agent 能提速，但没有成本和职责边界，容易变贵。
6. 文档过期后仍被 Agent 误用，导致上下文错误。

## 核心模型

Agent Context OS 将项目协作拆成九层：

| 层级 | 名称 | 作用 |
|---|---|---|
| 1 | 入口层 | 规定 Agent 如何开始工作 |
| 2 | 路由层 | 按任务类型决定读取哪些上下文 |
| 3 | 项目画像层 | 描述项目目标、非目标、用户和阶段 |
| 4 | 架构边界层 | 描述模块职责、依赖方向和禁止事项 |
| 5 | 业务语义层 | 描述业务对象、流程、字段、状态和异常 |
| 6 | 实现说明层 | 在开发前明确目标、方案、影响和验证 |
| 7 | 质量门禁层 | 用 workflow、checklist 和测试控制交付质量 |
| 8 | 历史问题层 | 沉淀问题根因、防复发规则和回归测试 |
| 9 | 成本控制层 | 控制 Token、上下文加载和多 Agent 触发 |

## 事实优先级

当文档、代码、测试和对话存在冲突时，默认按以下顺序判断：

```text
真实代码 / 测试结果
> 当前有效文档
> 历史文档
> 对话记录
> Agent 推测
```

Agent 发现文档与代码不一致时，不应直接忽略冲突，而应在任务报告中说明并建议更新文档。

## 通用性边界

核心蓝图只定义“如何描述项目和业务”，不内置任何具体项目的业务答案。

示例可以出现具体项目形态，但必须放在 `examples/` 下，并明确标注为示例。
