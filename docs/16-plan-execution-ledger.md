# 方案落实台账

## 目标

方案落实台账用于把用户与 Agent 讨论后的方案结论，整理成可确认、可排序、可执行、可验收的开发任务清单。

它不是原始会话记录，也不是当前项目事实源。它是方案从讨论走向开发的执行合同，用于防止讨论中逐步确认的细节在实现时丢失。

## 推荐位置

项目内推荐使用以下目录：

```text
docs/agent/plans/
├── README.md
└── <plan-id>.md
```

`docs/agent/intake.md` 负责登记方案入口、来源、阶段和影响范围；`docs/agent/plans/<plan-id>.md` 负责承载开发顺序、任务清单、确认点、验收口径和执行状态。

## 适用场景

- 用户围绕一个新方案进行了多轮讨论。
- 方案包含多个任务、多个模块或多个验收点。
- 用户要求整理开发顺序、形成任务清单、记录草稿或准备落地。
- 用户确认按完整方案开发，需要防止实现遗漏。

不适用场景：

- 一次性微小修改。
- 没有形成可执行任务的普通问答。
- 用户明确只想闲聊或比较思路，且不要求形成文档。

## 台账状态

| 状态 | 含义 | 使用规则 |
|---|---|---|
| `draft` | 根据讨论整理出的草稿 | 不能直接作为开发依据 |
| `confirmed` | 用户确认过的定稿 | 可作为实现依据 |
| `active` | 正在按序开发 | 必须维护任务状态 |
| `done` | 所有任务已完成或有明确处理结论 | 可触发最终文档同步 |
| `superseded` | 已被新台账替代 | 只用于回溯 |

## 任务状态

| 状态 | 含义 | 完成交付要求 |
|---|---|---|
| `pending` | 待开始 | 不得在最终交付中无说明地遗留 |
| `in_progress` | 正在处理 | 必须说明当前负责 Agent 或处理范围 |
| `done` | 已完成 | 必须关联验证结果或验收说明 |
| `blocked` | 被阻塞 | 必须记录阻塞原因和需要谁确认 |
| `deferred` | 用户确认延期或本轮不做 | 不视为遗漏，但必须记录原因 |

## 创建触发

Agent 不应在任意方案讨论中自动创建台账。

只有出现以下信号时，才允许创建 `draft` 台账：

- 用户要求“整理成方案”、“形成开发顺序”、“列任务清单”。
- 用户要求“记录草稿”、“准备落地”、“之后按这个做”。
- 讨论已经出现多个确认点，且用户同意将结论结构化保存。

仍处于 `discussion_only` 的普通讨论，只能在对话中整理候选方案、风险和待确认点，不得创建或更新台账。

## 标准结构

每份台账至少包含：

- 方案目标和非目标。
- 已确认点和待确认点。
- 开发顺序。
- 任务清单。
- 每个任务的来源摘要、依赖、验收口径和状态。
- 多 Agent 分工和回收结果。
- 完成核对。
- 变更记录。

任务清单必须使用稳定任务 ID，例如 `T1`、`T2`、`T3`。任务 ID 一旦进入 `confirmed`，不得复用给其他含义。

## 定稿规则

`draft` 转为 `confirmed` 前，Agent 必须让用户确认：

- 任务是否完整。
- 开发顺序是否正确。
- 哪些任务本轮不做。
- 每个任务是否有验收口径。
- 是否仍存在 `open` 待确认点。

未确认任务可以保留为 `open`，但不能混入 `confirmed` 任务中直接开发。

## 按序开发

当用户要求按完整定稿方案开发时，Agent 必须：

1. 读取 `confirmed` 或 `active` 台账。
2. 按开发顺序逐项执行任务。
3. 每完成一项，更新任务状态或在任务报告中列出状态变化。
4. 遇到 `blocked` 或 `deferred`，记录原因，不得静默跳过。
5. 最终报告列出任务总数、已完成、阻塞、延期和未开始数量。

只要仍存在没有原因的 `pending` 任务，就不得宣称完整方案已交付。

## 多 Agent 协作

完整方案落地时，主 Agent 应以台账为唯一任务来源来组织多 Agent：

- 主 Agent 负责任务顺序、任务分派、结果回收和最终整合。
- 子 Agent 只能处理主 Agent 指定的任务 ID 和范围。
- 子 Agent 输出必须回写到对应任务 ID 的状态、验证和风险。
- 主 Agent 必须逐项核对台账，不得只汇总子 Agent 的自然语言结论。

如果任务数量很少、强串行或用户要求单 Agent，主 Agent 可以不启用多 Agent，但必须在报告中说明原因，并仍按台账逐项执行。

## 文档同步时机

台账 `draft` 和 `confirmed` 阶段不得覆盖当前事实源。

只有任务完成并通过必要验证后，才把稳定事实写回：

- `docs/agent/memory.md`
- `docs/agent/style-profile.md`
- 业务文档
- 模块文档
- 决策记录
- 历史问题库或 checklist

台账状态为 `done` 时，Agent 必须评估是否需要最终文档同步；如果不需要，必须说明原因。

## 禁止事项

- 不得把原始会话全文保存为台账。
- 不得把 `draft` 台账作为实现依据。
- 不得在未确认开发顺序时开始完整方案实现。
- 不得静默跳过 `pending`、`blocked` 或 `open` 任务。
- 不得把子 Agent 的输出当作最终结论而不逐项核对台账。
- 不得在方案未落地前把草稿内容写成 `current`。
