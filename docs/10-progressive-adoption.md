# 渐进式接入系统

## 目标

渐进式接入系统用于让已经开发很久、文档结构不完整或已有旧文档体系的项目，以最低成本接入 Agent Context Engine。

首次接入的目标不是理解整个项目，而是安装协作入口、硬性读取规则、项目记忆和旧文档索引，让后续任务在真实开发过程中逐步完善上下文。

## 核心原则

1. **先挂载，后理解**：首次接入只建立协作规则，不全量分析代码和旧文档。
2. **最小可用**：只创建运行 Agent 协作所需的最小文件。
3. **按需迁移**：旧文档只在任务需要时读取、验证和迁移。
4. **不覆盖用户资产**：已有 `AGENTS.md`、`Agent.md`、文档和规则必须合并或备份，不得静默覆盖。
5. **事实分级**：旧文档默认不是当前事实，必须标记可信度和状态。
6. **动态完善**：每次任务结束都评估是否写回项目记忆、模块文档或旧文档索引。

## 接入模式

| 模式 | 适用项目 | 行为 |
|---|---|---|
| `fresh` | 新项目或几乎无历史包袱 | 可以直接复制完整模板 |
| `progressive` | 老项目、文档散乱或已有旧结构 | 只安装最小协作入口，后续按需补齐 |
| `mature` | 已完成主要上下文迁移 | 按完整 Agent Context Engine 规则运行 |

默认使用 `progressive`，除非用户明确要求完整初始化。

## 首次接入流程

Agent 在项目根目录收到“应用 Agent Context Engine”类请求时，应执行以下流程：

1. 检查是否存在 `AGENTS.md`、`Agent.md`、`docs/agent/`、旧文档目录和常见配置文件。
2. 向用户说明将修改或创建协作入口文件，并说明不会全量分析旧文档。
3. 等待用户确认。
4. 创建或合并最小协作文件。
5. 记录接入状态为 `progressive`。
6. 为旧文档建立轻量索引，不读取全文，不提炼业务规则。
7. 运行结构检查。
8. 告诉用户可以直接开始开发任务。

## 最小接入文件

```text
AGENTS.md
docs/agent/
├─ 00-index.md
├─ adoption.md
├─ memory.md
├─ legacy-docs.md
├─ quality.md
├─ task-report-template.md
├─ workflows/
│  ├─ progressive-adoption.md
│  ├─ new-feature.md
│  ├─ bug-fix.md
│  └─ refactor.md
└─ checklists/
   └─ adoption-checklist.md
```

项目可以后续按需补齐架构、技术栈、模块和业务语义文档。

## 旧文档处理策略

旧文档首次接入时只进入索引，不直接成为当前事实。

旧文档状态：

| 状态 | 含义 | 使用规则 |
|---|---|---|
| `unindexed` | 已发现但未建立索引 | 不作为上下文使用 |
| `indexed` | 已记录路径和主题 | 任务需要时可读取 |
| `reviewed` | 已对照代码或用户确认 | 可迁移有效事实 |
| `migrated` | 有效内容已写入当前文档 | 旧文档只用于回溯 |
| `deprecated` | 已确认过期 | 不作为实现依据 |

旧文档可信度：

| 可信度 | 含义 |
|---|---|
| `unknown` | 未确认 |
| `low` | 疑似过期或缺少来源 |
| `medium` | 与部分代码或用户描述一致 |
| `high` | 已被代码、测试或用户明确确认 |

## 按需迁移规则

只有出现以下情况，Agent 才应读取旧文档全文：

- 当前任务涉及旧文档对应模块或业务对象。
- 用户明确要求参考某份旧文档。
- 代码、测试和当前文档存在冲突，需要回溯历史。
- 修复复发问题，需要查看历史原因或旧方案。

读取后只迁移仍然有效、已确认、与当前任务有关的事实。

## AGENTS 合并规则

如果项目已有 `AGENTS.md` 或 `Agent.md`：

- 不得直接覆盖。
- 必须保留用户已有规则。
- 将 Agent Context Engine 规则追加到明确标记的区块。
- 如果已有规则和新规则冲突，必须报告冲突并等待用户确认。

推荐标记：

```md
<!-- BEGIN Agent Context Engine -->
<协作规则>
<!-- END Agent Context Engine -->
```

## 动态完善规则

接入后，每次任务结束前必须检查：

- 本次是否发现新的项目事实、业务事实或架构事实。
- 是否读取了旧文档。
- 旧文档状态是否需要从 `indexed` 更新为 `reviewed`、`migrated` 或 `deprecated`。
- 是否需要写回 `docs/agent/memory.md`。
- 是否需要补充模块文档、业务文档或历史问题库。

## 禁止事项

- 不得在首次接入阶段全量总结所有旧文档。
- 不得把旧文档内容直接标记为当前事实。
- 不得要求用户先填写完整模板才能开始开发。
- 不得静默覆盖用户已有协作规则。
- 不得为了追求文档完整性消耗大量 Token。
