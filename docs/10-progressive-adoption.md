# 老项目原地换芯升级

## 目标

老项目升级用于把已经接入旧版协作引擎的项目，切换到当前 Agent Context OS 引擎。目标不是长期兼容旧结构，而是原地换芯：

- 协作引擎规则换新。
- 有效项目记忆保留。
- 旧协作引擎文件清退。
- 用户项目保持极薄、干净、可维护。

## 核心原则

1. **引擎和记忆先拆分**：旧文件中属于协作规则的内容不迁移；属于项目事实的内容迁入项目记忆源。
2. **只保留一套当前结构**：升级后不长期保留旧 `docs/agent/`、旧 workflow、旧 checklist 或旧索引文档。
3. **项目极薄**：用户项目只保留 `AGENTS.md`、`.agent-context/config.json`、项目记忆源和检查脚本。
4. **本地索引可重建**：旧向量库、缓存或索引产物不迁移，升级后按新结构重建。
5. **业务事实可追溯**：迁入项目记忆源的内容必须有状态、可信度和证据路径。
6. **不假装懂业务**：未完成记忆源校验或索引重建时，不执行依赖完整业务上下文的高风险任务。

## 升级对象分类

| 旧内容 | 处理方式 |
|---|---|
| 旧版 AGENTS / Agent 入口规则 | 替换为新版极薄入口 |
| 旧版 workflow / checklist / 门禁规则 | 清退，由当前协作引擎统一提供 |
| 旧版项目背景、业务规则、交互习惯 | 提取为项目记忆源 |
| 旧版历史问题、防复发规则 | 提取为项目记忆源，并保留证据路径 |
| 旧版方案台账 | 若仍在执行，迁移为当前引擎台账格式；否则只保留必要记忆 |
| 旧版向量库 / 缓存 / 索引文件 | 不迁移，删除或忽略，按新结构重建 |

## 升级流程

1. 检查是否存在 `AGENTS.md`、`Agent.md`、旧 `docs/agent/`、旧协作文档和本地索引目录。
2. 识别旧内容的归属：协作引擎规范、项目共享记忆、个人偏好或废弃内容。
3. 将仍有效的项目共享记忆写入 `.agent-context/memory-sources/`。
4. 写入或替换新版 `AGENTS.md` 和 `.agent-context/config.json`。
5. 确认本地索引目录已被 `.gitignore` 排除。
6. 清退旧协作引擎文件，不长期保留两套结构。
7. 运行 `scripts/check-agent.ps1`。
8. 按当前引擎要求重建本地检索索引。
9. 执行必要验证并创建本地 Git 提交。

## 最小项目文件

```text
AGENTS.md
.agent-context/
├─ config.json
└─ memory-sources/
   ├─ README.md
   └─ *.jsonl
scripts/
└─ check-agent.ps1
```

## 记忆提取规则

只有以下内容可以进入项目记忆源：

- 当前仍有效的业务背景、业务规则、验收口径。
- 当前仍有效的用户交互习惯、端侧风格、团队开发习惯。
- 当前仍有效的模块边界、实现约束、验证命令。
- 已确认仍有价值的历史坑、防复发规则和证据路径。

以下内容不得进入项目记忆源：

- 旧版协作引擎的任务流程、门禁、报告格式。
- 已废弃或无法确认的历史方案。
- 无证据、无范围、无状态的长篇总结。
- 账号、密钥、token、cookie 或真实隐私数据。

## 升级后状态

升级完成后，用户项目应满足：

- 只有一套 Agent 入口。
- 没有旧协作引擎副本。
- 项目记忆源可 Git 管理、可评审、可合并。
- 本地索引不进入 Git，且可以重建。
- Agent 能通过当前协作引擎运行 S0-S3 门禁。

## 禁止事项

- 不得把旧协作引擎文档作为 legacy 长期保留在当前结构里。
- 不得把旧协作规则迁入项目记忆源。
- 不得把本地向量库文件提交到 Git。
- 不得为了“保留历史”复制一整套旧文档；需要回溯时使用 Git 历史。
- 不得在未校验项目记忆源时宣称已完成业务上下文迁移。
